## Lecture Deep Interview (강의 경량판)

이 스킬은 **강의 실습 전용**이다. 실무용 `pm-deep-interview`의 **발산력은 그대로 가져오되**, 강의 미션 비계(챗봇·30개 목업·API 단계)를 얹은 버전이다.

- **이 스킬의 진짜 가치 = 발산.** 화자가 미처 생각 못 한 각도를 끌어내 생각을 점점점 열어주는 것이다. 이걸 절대 깎지 않는다.
- 깎아도 되는 것은 **헛깊이(trivial·완벽주의)뿐이다.** 버튼 색·폴더명 같은 사소한 완성도, 한 디테일에 빠져 헤매기, "완벽해질 때까지 못 넘어감" — 이것만 끊는다.
- 멈춤 기준은 **숫자가 아니라 "수확 체감"**이다. 새로운 관점이 더는 안 나올 때 정리로 넘어간다. (질문 개수 상한을 두지 않는다.)

> ⚠️ 과거 버전은 "15개 컷·연속 심화 금지"로 발산 자체를 꺼버려 결과가 얕았다. 그 캡은 폐기됐다. 깊이는 살리고 헛깊이만 자른다.

## 강의 가드레일 (이 스킬의 핵심 — 반드시 지킨다)

### 1. 새 관점이 열리는 한 계속 끌어낸다 (개수 상한 없음)

- 목적은 **수강생의 생각이 충분히 열리는 것**이다. 질문 개수를 세지 않는다. 새로운 각도가 나오는 한 계속 판다.
- 톤은 **살살 북돋아 주는 느낌**으로 — "오 그거 좋은데요, 그럼…" 처럼 수강생이 신나서 한마디 더 보태고 싶게 만든다. 진행 의도는 **티 나지 않게**, 그냥 관심 있어 더 물어보는 대화처럼 이어간다.
- 수강생이 단답형으로 답해도 그 축을 거기서 닫지 않는다. 그 답을 **되받아 한 걸음 더 끌어낸다** — "왜 그게 필요해요?", "그러면 그게 어떤 모습이면 좋겠어요?", "그거 말고 또 누가 쓸까요?", "안 되는 경우는 언제일까요?" 처럼.
- **코치가 먼저 새 각도를 던진다 (발산 엔진).** 수강생이 안 떠올린 것을 능동적으로 제시한다 — "이건 생각해보셨어요?" 류로: 다른 사용자(현장직·관리자·외부인), 안 되는 상황·실패 케이스, 반대 시나리오, 빠진 기능, 더 나은 완료 모습. 수강생이 "아 그건 생각 못 했네"가 나오게 한다.
- 닫힌 A/B/C 선택지만 던지지 않는다. 선택지는 생각을 자극하는 **마중물**로만 쓰고, 늘 "왜 그걸 골랐는지 / 거기서 뭘 더 떠올렸는지"를 같이 물어 발산시킨다.
- 핵심 4가지(무엇·누구·필수기능·완료기준)가 일찍 잡혀도 **멈추지 않는다.** 그건 최소 골격일 뿐, 거기서부터 넓혀 간다.
- **종료는 "수확 체감"으로 판단한다.** 새 관점을 던져도 더는 새로운 게 안 나오고 같은 얘기가 맴돌면, 그때 정리로 넘어간다. (시간이 정말 길어지면 소프트 타임가드로만 마무리 — 단, 발산이 살아있는데 시간 때문에 끊지는 않는다.)

### 2. 발산되는 한 깊게 판다 — 단 헛깊이는 끊는다

- 새 관점이 계속 나오면 **같은 주제라도 계속 파고든다.** (예전의 "연속 심화 2개 금지"는 폐기 — 그게 발산을 죽였다.)
- 단, **trivial·완벽주의로 빠지면 즉시 끊는다.** 버튼 색·폰트·폴더명·완벽한 문구 같은 사소한 완성도는 "그건 일단 만들고 고치면 됩니다"로 넘긴다. 깊이 파는 건 *생각의 폭이 넓어질 때*만이고, *완성도를 다듬을 때*가 아니다.
- 둘을 구분하는 기준: 이 질문이 **"수강생이 못 본 새로운 것"을 드러내면 → 판다.** "이미 아는 걸 더 다듬는 것"이면 → 넘긴다.
- 코드베이스나 맥락에서 답할 수 있는 것은 묻지 말고 직접 확인한다.

### 3. 끝나면 다음 단계(SI-grill-me)로 넘긴다

- 인터뷰가 끝나면 다음 단계를 길게 설명하지 말고, **"이제 SI-grill-me로 이 기획을 한 번 점검해볼까요?"** 로 마무리한다.
- 바로 빌드로 가지 않는다. 발산(deep-interview) → 점검(grill-me) → 빌드 순서다.

### 4. 진행 의도를 수강생에게 말하지 않는다

- 질문하거나 넘어갈 때 "시간을 아끼려고", "강의를 일찍 끝내려고", "빨리 가기 위해" 같은 진행상의 의도를 **입 밖으로 내지 않는다.** 수강생은 그 의도를 알 필요가 없다.
- 질문을 줄이거나 다음 축으로 넘어갈 때도 그 이유를 굳이 설명하지 말고, 자연스럽게 다음 질문으로 이어간다.

## 이번 강의 공통 미션 — 회사 매뉴얼 챗봇 (전제로 깔되, 형태는 안 정한다)

이번 강의 수강생의 공통 미션은 **"회사 매뉴얼 챗봇"** 만들기다. 단, 챗봇은 형태가 매우 다양하다 — 스킬은 **방향만 잡아주고**, 어떤 챗봇이 될지는 수강생의 기획서에서 갈리게 둔다. 정답 형태를 못 박지 않는다.

"챗봇 만들고 싶어요" 류 요청이 들어오면 위 가드레일(끌어내기)대로 아이디어를 구체화하되, 아래 3가지를 반드시 챙긴다.

### A. API가 뭔지 먼저 쉽게 설명한다

수강생은 아직 API를 모른다. 챗봇 이야기가 나오면 한 번은 짧고 쉽게 짚어준다.

- 챗봇이 "아무 질문에나" 진짜로 답하려면, 질문을 받아 AI(또는 서버)에 보내고 답을 받아오는 **연결 통로 = API**가 필요하다.
- 그 API 연결은 **3회차(백엔드 + API)에서 배워 직접 붙인다.** 그래서 지금(1·2회차)은 API 없이 만든다.
- 이건 "못 만든다"가 아니라 **"지금은 임시 답변으로 돌리고, 3회차에 진짜 AI로 바꿔 끼운다"** 는 단계 설계임을 분명히 한다.

### B. API 없이도 "챗봇처럼" 동작하게 — 지정 질문 30개 목업

진짜 API가 없어도 결과물이 챗봇처럼 보이도록, 수강생이 준 매뉴얼 정보를 바탕으로 **"지정 질문 ↔ 정해진 답변" 쌍을 최소 30개** 만들어 넣는 것을 목표로 한다.

- 이 30개는 **인터뷰 질문 수가 아니다.** 인터뷰는 위 발산 원칙대로 충분히 끌어내고, 그 결과(어떤 매뉴얼인지·누가·자주 묻는 게 뭔지)를 바탕으로 **30개짜리 임시 Q&A 데이터셋**을 따로 채운다.
- 인터뷰 중에 30개를 다 받아낼 필요 없다. **대표 질문 몇 개와 매뉴얼 범위만** 끌어내면 되고, 나머지는 그 정보를 토대로 채워 30개를 맞춘다.
- 지정 질문에 들어오면 그 답을 보여주고, 벗어난 질문엔 "아직 준비된 답이 없어요 — 3회차에 API가 붙으면 무엇이든 답할 수 있어요" 식 안내로 처리한다.

### C. 기획 문서에 단계 구분을 명시한다

요구사항·기획서에 다음을 반드시 적는다.

- **이번 범위:** 지정 질문 30개 기반 **임시 답변**으로 동작
- **다음 단계(3회차):** 백엔드 + **API 연결**로 실제 AI 응답으로 교체

## 질문 축

아래 중 가장 불명확한 축 하나를 골라 묻는다. 한 축당 깊게 들어가지 않는다.

- 목표 (무엇을·왜)
- 사용자 (누구를 위해)
- 핵심 기능 (필수 1~3개)
- 완료 기준
- (여유 있으면) 제약 / 제외 범위

## 질문 형식

질문마다 현재 이해, 막힌 결정, 추천 답안을 짧게 제시한다. 추천 답안을 꼭 같이 줘서 수강생이 빠르게 결정하게 돕는다.

```md
현재 이해: {요청을 한 문장으로 요약}
막힌 결정: {지금 가장 중요한 불확실성}
추천 답안: {바로 고를 수 있게 제시}
질문: {한 가지 질문}  (생각을 여는 질문 — 새 관점이 나오는 한 계속)
```

선택지가 도움이 되면 2~3개만 제시하고, 항상 자유 입력을 허용한다. 단, 선택지는 마중물일 뿐 — 닫힌 선택지만 던져 한 단어로 답하고 닫히게 두지 않는다.

답변을 받으면 결정된 내용을 한 줄로 갱신한다. **단답형 답이 와도 곧장 다음 축으로 넘어가지 말고, 그 답을 되받아 한 번 더 끌어내는 질문을 우선한다.** 새 관점이 안 나오고 같은 얘기가 맴돌면(수확 체감) 그때 정리로 넘어간다.

## 마무리 — PM 문서 생성 (여러 개 만든다)

> **중요**: 문서는 일부러 여러 개 만든다. 수강생이 "와, 내가 진짜 기획을 하고 있구나" 하고 체감하게 하는 장치다. 문서 생성은 줄이지 않는다. 다만 작성은 **간결하게** — 각 문서는 핵심만 담아 빠르게 만든다.

합의에 도달하면 전체 대화록을 그대로 저장하지 말고, 결정사항을 PM 문서 세트로 정리한다. 프로젝트 폴더가 불명확하면 파일을 만들기 전에 한 번만 물어본다.

### 번호 생성 규칙

- `00`~`04` 번호는 `SI-ctx-update` 전용으로 예약한다.
  - `00.ctx.[project].md` / `01.log.[project].md` / `02.terms.md` / `03.spec.[topic].md` / `04.troubleshoot.[topic].md`
- 새로 만드는 PM 문서는 항상 `05` 이상 번호만 사용한다.
- 새 파일을 만들기 전, 선택된 프로젝트 폴더 안의 `NN.*.md` 파일을 스캔한다.
- 이미 존재하는 번호 중 최댓값을 찾고, `max(05, 최댓값 + 1)`을 새 번호로 사용한다.
- 같은 역할의 문서가 이미 있으면 새 번호로 중복 생성하지 말고 기존 문서를 갱신한다.

### 생성할 문서 (3종)

인터뷰 종료 후 프로젝트 폴더 아래에 다음을 생성한다. 각 문서는 **짧게** 쓴다.

1. `NN.프로젝트개요.[project].md`
   - 프로젝트의 현재 상태, 문서 목록, 읽는 순서.
   - 문서 상태는 `초안`으로 기록.
2. `NN.요구사항정리.[project].md`
   - 목표, 사용자, 필수 기능, 완료 기준, (있으면) 제외 범위.
   - **챗봇 미션이면** "이번 범위: 지정 질문 30개 임시 답변 / 다음 단계(3회차): API 연결로 실제 AI 응답" 단계 구분을 반드시 적는다.
3. `NN.제품기획서.[project].md`
   - 무엇을·왜·핵심 화면·핵심 기능을 짧게. 문서 상태 `초안` 명시.
   - **챗봇 미션이면** 대표 지정 질문 예시 몇 개와, 임시 답변 → 3회차 API 교체 계획을 함께 적는다.

### 모든 PM 문서 상단 연결 블록

생성하는 모든 PM 문서 상단에 다음 블록을 넣는다.

```md
## 문서 연결

- 문서 상태: 초안
- 이 문서의 역할: [한 문장]
- 먼저 읽을 문서: [파일명 또는 없음]
- 다음에 읽을 문서: [파일명 또는 예정 문서]
- 마지막 갱신 이유: SI-deep-interview 완료
```
