# SI-ui-design-system — 코드 전 UI 디자인 시스템 설계

## 목적

이 스킬은 실제 코드를 수정하지 않는다. 디자인 레퍼런스를 읽고, 구현자가 나중에 일관된 UI를 만들 수 있도록 PM 문서 세트 안에 `NN.기술설계.UI디자인시스템.[project].md`를 생성 또는 갱신한다.

`SI-shadcn`이 실제 Next.js + Tailwind + shadcn/ui 코드에 디자인을 적용하는 구현 스킬이라면, `SI-ui-design-system`은 그 전에 디자인 기준을 문서화하는 설계 스킬이다.

## 언제 사용하나

다음 상황에서 사용한다.

- 사용자가 `DESIGN.md`, 브랜드가이드, 무드보드, 스크린샷, 참고 URL을 주고 디자인 기준을 정리하라고 할 때
- 코드 구현 전 UI 톤, 색상, 버튼, 카드, 표, 배지, 다이얼로그 규칙을 먼저 정해야 할 때
- 강의용으로 디자인 설계 단계와 실제 구현 단계를 분리해서 보여주고 싶을 때
- 나중에 `SI-shadcn`이 참고할 기준 문서를 만들고 싶을 때

## 하지 않는 것

- Next.js 프로젝트 생성 안 함
- `npx shadcn init` 실행 안 함
- `components.json`, `globals.css`, Tailwind 설정 수정 안 함
- React 컴포넌트 구현 안 함
- 실제 UI 리팩터링 안 함

## 입력 우선순위

1. 사용자가 직접 지정한 디자인 기준 파일 또는 이미지 또는 URL
2. 프로젝트 폴더의 `DESIGN.md`, `디자인*.md`, `*style*.md`
3. 기준이 없으면 사용자에게 한 번만 확인

## 산출물 파일명

선택된 프로젝트 폴더 아래에 저장한다.

```md
NN.기술설계.UI디자인시스템.[project].md
```

번호 규칙:

- `00`~`04`는 예약 번호로 사용하지 않는다.
- 기존 `NN.기술설계.UI디자인시스템.*.md`가 있으면 새로 만들지 말고 갱신한다.
- 없으면 프로젝트 폴더의 기존 `NN.*.md` 중 최댓값 다음 번호를 사용한다.

## 문서 상단 연결 블록

생성 또는 갱신하는 문서에는 다음 블록을 넣는다.

```md
## 문서 연결

- 문서 상태: 검토중 | 확정
- 이 문서의 역할: UI 디자인 기준과 컴포넌트 사용 규칙을 정리한다.
- 먼저 읽을 문서: 프로젝트개요, 요구사항정리, 제품기획서, 기획검토, DESIGN.md
- 다음에 읽을 문서: 작업목록, 구현계획, SI-shadcn 적용 문서
- 이 문서를 참고할 때: 화면 구현, shadcn 테마 적용, UI 리팩터링을 할 때
- 마지막 갱신 이유: SI-ui-design-system으로 디자인 기준 정리
```

## 문서에 포함할 내용

### 1. 디자인 기준 출처

- 사용한 파일 또는 이미지 또는 URL
- 레퍼런스에서 그대로 가져올 것
- 프로젝트 특성 때문에 조정할 것
- 가져오지 않을 것

### 2. 디자인 토큰 요약

표로 정리한다.

| 영역 | 기준 |
|---|---|
| 색상 | 배경, 표면, 텍스트, 강조, 경고, 보더 |
| 타이포그래피 | 본문, 제목, 라벨, 메타 정보 |
| 모서리 | 버튼, 카드, 패널, 다이얼로그 |
| 여백/밀도 | 대시보드 밀도, 카드 간격, 표 밀도 |
| 그림자 | 사용 여부 |
| 무드 | 한 줄 설명 |

### 3. shadcn/ui 컴포넌트 후보

코드 구현 전 기준만 정한다.

예:

| 화면 요소 | 나중에 쓸 shadcn 후보 | 사용 기준 |
|---|---|---|
| 저장/추가/삭제 버튼 | Button | 각진 버튼, inline style 금지 |
| 프로그램 현황 박스 | Card | 일반 카드 8px radius, 그림자 없음 |
| 전체 프로그램 목록 | Table | PC 대시보드 기준 밀도 유지 |
| 상태값 | Badge 또는 텍스트 라벨 | 알약형 남발 금지 |
| 새 프로그램 추가 | Dialog | 작은 폼 팝업 |
| 상태 선택 | Select | 상태값 변경 |
| 성공/실패 알림 | Toast 또는 Sonner | 저장 결과 안내 |

### 4. 화면별 UI 기준

프로젝트에 필요한 화면별로 정리한다.

- 로그인 화면
- ERP 기본 레이아웃
- 대시보드
- 프로그램 관리
- 할 일/TODO
- 회의 입력
- 회의 분석 결과
- 확장 예정 메뉴

각 화면마다 다음을 적는다.

- 화면 목적
- 주요 UI 요소
- 권장 컴포넌트
- 디자인 주의점
- 1차에서 하지 않을 것

### 5. 금지 규칙

프로젝트 UI 품질을 흔드는 요소를 명시한다.

예:

- inline style 사용 금지
- 순수 흰색 배경 금지
- 그림자 남발 금지
- 둥근 알약 배지 남발 금지
- 랜딩페이지식 hero 남발 금지
- 2차 기능을 실제 기능처럼 보이게 만들지 않기

### 6. SI-shadcn으로 넘길 내용

나중에 실제 구현 단계에서 `SI-shadcn`에게 줄 요약을 마지막에 둔다.

```md
## SI-shadcn 적용 메모

- 디자인 기준 파일: [파일명]
- 테마 방향: [한 줄]
- 우선 적용 화면: [화면명]
- 필요한 shadcn 컴포넌트: [목록]
- 구현 시 주의: [주의점]
```

## 진행 방식

1. 지정된 디자인 레퍼런스를 읽는다.
2. 프로젝트의 `프로젝트개요`, `요구사항정리`, `제품기획서`, `기획검토`를 필요한 만큼 읽는다. 이미 `작업목록`이 있으면 함께 읽되, 없어도 진행한다.
3. 디자인 토큰을 추출한다.
4. 프로젝트 성격에 맞춰 조정할 것과 버릴 것을 구분한다.
5. `NN.기술설계.UI디자인시스템.[project].md`를 생성 또는 갱신한다.
6. 프로젝트개요가 있으면 새 기술설계 문서 링크와 현재 단계를 `SI-ui-design-system 완료`로 반영한다.
7. 제품기획서나 작업목록이 이미 있으면 `참고 UI 디자인 시스템` 링크를 추가한다. 없으면 이후 `SI-to-issues`와 `SI-write-plan`이 이 문서를 먼저 읽도록 안내한다.
8. 저장 후 생성 또는 갱신한 파일 경로를 알려준다.

## 초보자용 설명 원칙

사용자에게 설명할 때는 다음 비유를 쓴다.

- `SI-ui-design-system`: 인테리어 설계도
- `SI-shadcn`: 실제 시공
- `DESIGN.md`: 참고 사진 또는 무드보드
- shadcn 컴포넌트: 문, 창문, 책상 같은 표준 부품

`SI-shadcn`을 지금 꼭 실행해야 한다고 말하지 않는다. 코드 프로젝트가 없으면 `SI-ui-design-system`까지만 진행한다.
