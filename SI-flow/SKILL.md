# SI-flow — 개발 워크플로우 가이드

SI- 스킬 워크플로우를 개발 순서대로 안내한다. 각 스킬이 끝나면 다음 단계를 추천한다. 초보 개발자도 이 흐름대로 따라오면 기획부터 구현 준비까지 같은 프로젝트 폴더 안에서 문서가 이어진다.

## 전체 순서

| 단계 | 스킬 | 성격 | 역할 |
|------|------|------|------|
| 1 | `/SI-caveman` | 필수 | 토큰 절감 모드 ON — 항상 첫 번째 |
| 2 | `/SI-auto` | 선택 | 아이디어 윤곽이 아예 없을 때만 |
| 3 | `/SI-deep-interview` | 필수 | 요구사항 발산 + 프로젝트개요/요구사항정리/제품기획서 초안 |
| 4 | `/SI-grill-me` | 필수 | 기획검토 + 제품기획서/프로젝트개요 갱신 |
| 5 | `/SI-office-hours` | 선택 | 막혔을 때 돌파구 |
| 6 | `/SI-ui-design-system` | 선택 | 코드 전 DESIGN.md 기반 UI 디자인 시스템 문서 생성 |
| 7 | `/SI-to-issues` | 선택 | UI 기준을 참고해 실행 가능한 세로 조각을 작업목록으로 정리 |
| 8 | `/SI-write-plan` | 필수 | 각 작업을 UI 기준까지 포함한 구현계획으로 구체화 |
| 9 | `/SI-shadcn` | 선택 | Next.js 프로젝트에 shadcn/ui 화면 구현·리팩터링 |
| 10 | `/SI-requesting-code-review` | 선택 | 코드 검토 |
| 11 | `/SI-ctx-update` | 필수 | 마무리 — ctx 정리 |

## 동작 방식

### 시작 시
`/SI-flow` 를 실행하면:
1. 전체 순서표를 보여준다
2. 1단계 `/SI-caveman` 을 즉시 실행한다

### 각 단계 완료 후
스킬 실행이 끝나면 항상 이 형식으로 다음을 안내한다:

```
✅ [N단계: 스킬명] 완료
다음 → [N+1단계: 스킬명] ([성격]) — [한 줄 설명]
진행할까요? (선택 단계는 건너뛸 수 있습니다)
```

### 선택 단계 처리
- 선택 단계는 "건너뛸게요" 또는 "skip" 하면 바로 다음 필수 단계로 넘어간다
- 선택 단계도 필요하면 언제든 개별 호출 가능하다

## 각 스킬 한 줄 설명

- **SI-caveman**: AI 답변을 원시인처럼 압축 — 토큰 절감, 핵심만 남김
- **SI-auto**: 아이디어 없을 때 AI가 기획 초안을 자동 생성
- **SI-deep-interview**: 요구사항을 발산하고 `프로젝트개요`, `요구사항정리`, `제품기획서` 초안을 프로젝트 폴더에 저장
- **SI-grill-me**: 기획의 약점을 검토하고 `기획검토`를 생성, 바뀐 내용은 `프로젝트개요`와 `제품기획서`에 반영
- **SI-office-hours**: 막히는 지점에서 방향을 잡아주는 코칭 세션
- **SI-ui-design-system**: 코드 구현 전 `DESIGN.md` 같은 레퍼런스를 읽고 `기술설계.UI디자인시스템` 문서 생성
- **SI-to-issues**: 기획과 UI 디자인 기준을 독립 실행 가능한 세로 조각으로 나누고 `작업목록`을 프로젝트 폴더에 저장
- **SI-write-plan**: `작업목록`의 각 작업을 UI 기준, 파일 경계, 테스트, 명령어까지 포함한 `구현계획`으로 구체화
- **SI-shadcn**: Next.js + Tailwind 프로젝트가 생긴 뒤 shadcn/ui 테마와 컴포넌트로 실제 화면 구현·리팩터링
- **SI-requesting-code-review**: 완성된 코드 품질 검토
- **SI-ctx-update**: 오늘 세션 작업 내용을 ctx 파일에 정리

## 프로젝트 폴더 파일 번호 규칙

워크플로우가 만드는 PM 문서는 선택된 프로젝트 폴더 아래에 번호 접두사로 정렬된다.

```
00 ~ 04  → SI-ctx-update 계열 예약 (00.ctx / 01.log / 02.terms / 03.spec / 04.troubleshoot)
05 ~     → PM 문서 자동 번호 생성
```

새 PM 문서를 만들 때는 프로젝트 폴더의 기존 `NN.*.md` 파일을 스캔하고, `max(05, 기존최대번호 + 1)`을 사용한다. 같은 역할 문서가 이미 있으면 새 번호로 중복 생성하지 말고 기존 문서를 갱신한다.

PM 문서 세트:

```md
NN.프로젝트개요.[project].md
NN.요구사항정리.[project].md
NN.제품기획서.[project].md
NN.기획검토.[project].md
NN.기술설계.[project].md
NN.기술설계.UI디자인시스템.[project].md
NN.실행계획.[project].md
NN.작업목록.[project].md
NN.구현계획.[issue-or-feature].md
NN.결정기록.[project].md
```

`프로젝트개요`는 전체 문서 흐름의 현재 지도다. `SI-grill-me`, `SI-ui-design-system`, `SI-to-issues`, `SI-write-plan`이 끝날 때마다 새 문서 링크와 현재 단계를 갱신한다.

## 구현 단계 문서 묶음

최종 구현 AI에게는 전체 문서를 모두 주지 말고 다음 묶음을 우선 전달한다.

1. `프로젝트개요`
2. `작업목록`의 해당 작업
3. `구현계획`
4. `기술설계`
5. `기술설계.UI디자인시스템`
6. 필요 시 `제품기획서`, `기획검토`, `결정기록`

## 절대 하지 않는 것

- 사용자가 "건너뛰겠다"는데 선택 단계를 강요하는 것
- 스킬 실행 없이 설명만 하고 넘어가는 것
- 다음 단계 안내 없이 끝내는 것 (SI-ctx-update 완료 전까지)
