이번 세션에서 작업한 프로젝트의 ctx 파일을 찾아서 업데이트한다.

## 완료 조건

- ctx 저장 + **세션 회고** + 길이 검사 + 완료 보고까지 끝나야 완료.
- 세션 회고를 생략한 채 "ctx 업데이트 완료"라고 보고하면 안 된다 (반영 사항이 없으면 "회고: 특이사항 없음"이라고 명시).

## ctx 파일 명명 규칙

모든 ctx 파일은 번호 접두사를 사용한다. 폴더 정렬 기준:

```
00.ctx.[project].md        — 현재 상태 + 다음 할 일 (50~80줄 목표, 100줄 상한)
01.log.[project].md        — 세션 작업 요약 누적
02.terms.md                — 도메인 용어 사전
03.spec.[topic].md         — 세부 기술 스펙·기준
04.troubleshoot.[topic].md — 시행착오·문제 해결 기록
```

ctx 내부에 `## 동반 파일` 섹션을 **맨 위**에 두고, 각 파일에 트리거 조건을 명시한다:
```
## 동반 파일
- `01.log.[project].md` — 최근 세션 작업 파악 시 Read
- `02.terms.md` — 도메인 용어 헷갈릴 때 Read
- `03.spec.[topic].md` — [구체적 작업 키워드] 작업 시 Read
- `04.troubleshoot.[topic].md` — [구체적 증상 키워드] 발생 시 Read / 같은 에러 2회 반복 시 3번째 시도 전 Grep
```

### 트리거 조건 작성 규격

매칭 가능한 **구체적 키워드**만 허용한다:
- ❌ 금지: "관련 문제 시 Read", "필요할 때 Read" (모호해서 실제로 안 읽힘)
- ✅ 강제: "Notion API 호출 작업 시", "빌드 에러 ENOENT 발생 시", "git push 한글 파일명 작업 시"
- `04.troubleshoot` 링크에는 표준 트리거를 반드시 포함: **"같은 에러 2회 반복 시 3번째 시도 전 Grep"**

### 04.troubleshoot 항목 포맷 (상태 표기 필수)

```
- [가설] git push 시 한글 파일명 깨짐 → core.quotepath 의심 (2026-06-13, 1회)
- [검증됨] PowerShell+Notion API는 UTF-8 변환 필수 (05-12 발생, 06-01 재확인)
```

- `[가설]` = 1회 관찰. 아직 규칙 아님.
- `[검증됨]` = 같은 패턴 2회 이상 재현. "X 상황이면 Y 하라" 형태의 한 줄로 정제해서 기록.

### ctx 인라인 주의사항 섹션 (참조 1겹)

검증된 규칙 중 핵심은 ctx 안에 직접 둔다 — ctx는 프로젝트 진입 시 무조건 읽히므로 추가 토큰 비용이 없다:

```
## ⚠ 검증된 주의사항
- PowerShell+Notion → UTF-8 변환 필수 (상세: 04.troubleshoot §인코딩)
```

- **최대 5줄.** 초과 시 가장 덜 중요한 항목을 `04.troubleshoot`로 강등.
- 각 줄은 한 줄 규칙 + 상세 위치 포인터 형식.
- `[검증됨]` 항목만 올릴 수 있다. `[가설]`은 올리지 않는다.

## 동작 순서

1. **프로젝트 감지** — 세션 대화 맥락 기준으로 어떤 프로젝트를 작업했는지 파악. `C:\Users\mshse\.claude\CLAUDE.md`의 키워드 표에서 해당 ctx 파일 경로 찾기.

2. **ctx 파일 Read 또는 생성**
   - 기존 ctx 파일이 있으면 파일 전체를 읽어 현재 섹션 구조 파악.
   - 작업한 프로젝트 폴더가 명확하지만 ctx 파일이 없으면, `00.ctx.[project].md`를 새로 만든다.
   - 필요성이 명확하면 `01.log`, `02.terms`, `03.spec`, `04.troubleshoot`도 판단해서 생성할 수 있다.
   - 동반 파일을 생성했으면 `00.ctx`의 `## 동반 파일` 섹션에 파일명과 Read 트리거 조건을 반드시 기록한다.
   ```
   # ctx. [project]

   > 마지막 업데이트: YYYY-MM-DD

   ## 동반 파일

   ## ⚠ 검증된 주의사항

   ## 현재 상태
   - [이번 세션 완료 내용 1~2줄]

   ## 다음 할 일
   - [ ] [다음 행동]

   ## 프로젝트 개요
   - 목적: [짧게]
   ```

3. **세션 분석** — 이번 대화에서 완료된 작업, 새로 생긴 할 일, 변경된 상태 파악.

4. **3가지만 갱신** (나머지 섹션은 절대 수정 금지 — 단, 5번 회고 결과는 `## ⚠ 검증된 주의사항`과 `04.troubleshoot`에 반영 가능):
   - `마지막 업데이트` → 오늘 날짜 (YYYY-MM-DD)
   - `현재 상태` → 이번 세션 완료 내용 반영 (1~2줄, 간결하게)
   - `다음 할 일` → 완료된 항목 제거 + 새 항목 추가

5. **세션 회고** — 이번 세션을 3가지 관점으로 스캔한다. 해당 사항이 없으면 "회고: 특이사항 없음" 한 줄로 끝낸다 (억지로 만들지 않는다):
   - 실수·실패와 그 원인 (에러, 잘못된 접근, 되돌린 작업)
   - 사용자가 교정한 피드백
   - 새로 확정된 작업 방식

   **참조 실패 점검** — 이번 세션 에러 중 답이 이미 `04.troubleshoot`에 있었던 것이 있으면, 그건 '참조 실패 사건'이다. 해당 항목을 ctx `## ⚠ 검증된 주의사항`으로 승격하고 동반 파일 트리거 조건을 더 구체적인 키워드로 고친다.

   **학습 기록 규칙**:
   - 처음 관찰된 실패 → `04.troubleshoot`에 `[가설]`로 기록 (날짜·1회 표시)
   - 기존 `[가설]`과 같은 패턴 재발 → `[검증됨]` 승격 + "X 상황이면 Y 하라" 한 줄로 정제
   - 일회성 단순 실수(오타 등)는 기록하지 않는다 — 반복됐거나 사용자가 직접 교정한 것만

   **라우팅 기준** (어디에 남기나):

   | 학습의 성격 | 보낼 곳 |
   |---|---|
   | 이 프로젝트 한정 기술 문제 | `04.troubleshoot.[topic].md` |
   | 프로젝트 무관, 작업 방식 전반 | Claude 메모리 |
   | 스킬 자체의 결함·개선 | 수정안 제시 → 사용자 승인 후 해당 스킬 파일 반영 |

   **프루닝** (누적 방지):
   - `## ⚠ 검증된 주의사항`은 최대 5줄 — 초과 시 가장 덜 중요한 항목을 `04.troubleshoot`로 강등
   - `04.troubleshoot`의 `[가설]` 중 6개월간 재발·참조 없는 항목은 삭제 제안

6. **길이 검사 + 분리 제안** — 저장 후 ctx 줄 수 확인. **100줄 초과 AND 아래 섹션 유형 감지 시** 한 번만 제안:

   | 감지 패턴 | 분리 대상 파일 |
   |-----------|--------------|
   | 날짜 있는 작업 요약 블록 | `01.log.[project].md` |
   | 도메인 용어 정의 나열 | `02.terms.md` |
   | 세부 기술 스펙·청킹 기준·검색 구조 등 | `03.spec.[topic].md` |
   | 문제 해결 기록·시행착오·디버깅 로그 | `04.troubleshoot.[topic].md` |

   제안 형식: "ctx가 XXX줄입니다. `[섹션명]` → `[파일명]`으로 분리할까요?"
   - 분리 필요성이 명확하면 사용자 확인 없이 분리 실행 + ctx에 `## 동반 파일` 링크 추가 (트리거 조건 포함)
   - 애매하면 한 번만 사용자에게 확인

7. **보고** — "✅ ctx 업데이트 완료 → [파일경로] / 회고: [반영 내용 또는 특이사항 없음]" 한 줄로 끝.

## 절대 하지 않는 것

- 프로젝트 목표, 아키텍처, 링크, 기타 고정 섹션 수정
- 내용을 길게 늘리는 것 (ctx는 항상 가볍게 유지)
- 불필요한 동반 파일을 습관적으로 생성
- 생성한 동반 파일을 `00.ctx`의 `## 동반 파일`에 연결하지 않고 방치

## 예외 처리

- **프로젝트 폴더를 특정 못 하면** → "어떤 프로젝트 ctx를 업데이트할까요?" 한 번만 묻고 진행
- **프로젝트 폴더는 명확하지만 ctx 파일이 없으면** → 묻지 말고 `00.ctx.[project].md`를 생성한 뒤 이번 세션 상태를 기록
- **로그·용어·스펙·트러블슈팅 분리가 필요하면** → `01~04` 규칙에 맞춰 파일을 생성하고 `00.ctx`에 동반 파일 링크와 트리거 조건을 기록
- **이번 세션에 특별한 변경 없으면** → 날짜만 업데이트하고 나머지 유지
- **해당 섹션이 ctx에 없으면** → 없는 섹션은 추가하지 않고 있는 것만 갱신 (단, 회고에서 `[검증됨]` 규칙이 처음 나오면 `## ⚠ 검증된 주의사항` 섹션은 생성 가능)
- **ctx가 100줄 이하이거나 분리 가능 섹션 없으면** → 길이 검사 결과 보고하지 않음
