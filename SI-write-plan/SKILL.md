# SI-write-plan (강의 경량판)

이 스킬은 **강의 실습 전용**이다. 실무용 `pm-write-plan`과 목표가 다르다.

- 실무판: TDD 루프(실패 테스트 → 구현 → 통과 → 커밋)를 모든 작업마다, 모든 단계에 완전한 코드. 시간 무제한.
- 이 강의판: **"무엇을, 어떤 순서로 만들지" 한눈에 보이는 짧은 체크리스트.** 바로 만들기 시작할 수 있게.

수강생이 계획서 쓰다가 정작 앱을 못 만드는 일을 막는다. 계획은 **빌드로 가는 발판**이지 목적이 아니다.

---

## 강의 가드레일 (이 스킬의 핵심 — 반드시 지킨다)

### 1. TDD 루프를 강요하지 않는다

- "실패하는 테스트 작성 → 실행 → 구현 → 다시 실행 → 커밋" 5단계 반복 **금지**.
- 강의 앱은 바이브코딩으로 만든다. 테스트 먼저 짜라고 하지 않는다.
- 필요하면 "다 만든 뒤 직접 눌러보며 확인"으로 충분하다.

### 2. 계획은 짧게 — 한 화면 안에 들어오게

- 단계를 2~5분짜리로 잘게 쪼개지 않는다. **"화면/기능 단위"로 묶어** 5~8줄 체크리스트면 충분하다.
- 모든 단계에 완전한 코드 붙이기 **금지**. 코드는 Claude Code가 빌드할 때 만든다. 계획서엔 "무엇을 만들지"만.
- 파일 경로·컴포넌트 이름을 정확히 못 적어도 된다 — 만들면서 정해도 된다.

### 3. 끝나면 바로 빌드로 직결

- 계획이 나오면 길게 설명하지 말고 **"이 순서대로 바로 만들러 갈까요?"** 로 마무리한다.
- subagent 실행 위임, executing-plans 같은 무거운 실행 체계로 넘기지 않는다. 그냥 Claude Code에게 만들어 달라고 한다.

## 계획서 형식 (짧게)

```md
# [앱/기능 이름] 만들기 계획

목표: [한 문장 — 무엇이 되면 끝인가]
화면: [핵심 화면 2~4개]

## 만드는 순서
- [ ] 1. [첫 화면/기능] — [한 줄 설명]
- [ ] 2. [다음] — [한 줄 설명]
- [ ] 3. [다음] — [한 줄 설명]
- [ ] 4. 직접 눌러보며 확인
- [ ] 5. Vercel 배포

## 디자인 기준
- [UI 디자인 시스템 문서가 있으면 그 파일명, 없으면 "기본" ]
```

- 순서는 **"눈에 보이는 결과가 가장 빨리 나오는 것"부터**. 첫 단계가 끝나면 화면에 뭔가 떠야 한다.
- 큰 앱이면 `SI-to-issues`로 먼저 조각낸 뒤, 조각 하나마다 이 계획을 짧게 만든다.

## 저장 규칙 (PM 문서 유지)

> 문서 생성은 유지한다. 수강생이 "내가 계획을 세우고 만든다"를 체감하는 장치다. 다만 내용은 위 형식처럼 **짧게**.

- 결과는 선택된 프로젝트 폴더에 `NN.구현계획.[기능].md`로 저장한다.
- `00`~`04`는 `SI-ctx-update` 예약 번호. 새 문서는 `05` 이상, 프로젝트 폴더의 `NN.*.md` 최댓값 다음 번호를 쓴다.
- 같은 기능의 구현계획이 이미 있으면 새로 만들지 말고 갱신한다.
- `C:\Users\mshse\Desktop\ClAgent` 루트에는 만들지 않는다.
- 저장 후 파일 경로를 보여주고 **"이 순서대로 바로 만들러 갈까요?"** 로 마무리한다.
